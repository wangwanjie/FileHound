## Context

FileHound 已经有 Sparkle 运行时代码、更新偏好设置页和发布脚本，但这些部分现在没有形成一条可靠的端到端链路。发布产物中实际缺少稳定的 `SUFeedURL` / `SUPublicEDKey`，`appcast` 后处理可能丢失 Sparkle 依赖的签名字段，更新设置页与应用主菜单也没有统一的更新触发入口。与此同时，正式发布仍默认产出 Universal 包，无法按 `arm64` 与 `x86_64` 分架构发布和分架构更新。

这次变更同时跨越应用运行时、偏好设置 UI 和发布脚本，属于典型的跨模块改造。设计必须明确三个边界：

- 应用内由谁负责统一更新能力状态和动作
- 偏好设置页如何消费该状态并反映给用户
- 发布链路如何按架构生产与消费 Sparkle feed

## Goals / Non-Goals

**Goals:**
- 修复 Sparkle 更新链路，让手动检查更新和启动时检查更新都具备可用前提。
- 让设置页和主菜单共享同一套更新状态与触发逻辑。
- 把正式发布流程改成 `arm64` / `x86_64` 分架构构建、分架构 feed、分架构资产。
- 保留当前“分步脚本、人工可检查”的发布方式。

**Non-Goals:**
- 不尝试让已经缺失关键 Sparkle 元数据的历史安装包自动恢复更新。
- 不把发布流程改造成一键式总脚本或 CI 自动化流水线。
- 不引入新的更新框架或替换 Sparkle。

## Decisions

### 1. 由 `UpdateManager` 统一暴露更新状态与动作

`UpdateManager` 继续作为应用内更新协调者，但职责从“简单包一层 Sparkle”提升为“统一状态来源”：

- 统一计算当前是否可检查更新
- 统一执行手动检查更新
- 统一同步启动时检查和自动下载等策略到 Sparkle runtime

这样设置页按钮和主菜单不会再各自维护一套判断逻辑。

备选方案：
- 继续让偏好设置页和主菜单分别直连 Sparkle
  - 放弃，因为状态容易漂移，用户会看到一个入口可点、另一个不可点

### 2. 更新偏好设置页只消费 `UpdateManager` 状态，不复制更新判断

更新设置页中的“立即检查更新”按钮应当根据 `UpdateManager` 的状态启用或禁用，并在不可用时给出明确原因提示。偏好设置本身只负责持久化用户策略，不再重复判断 feed 是否完整或 Sparkle 是否可启动。

备选方案：
- 在视图控制器里继续直接读 `Info.plist` 键值
  - 放弃，因为这会把运行时逻辑散落到 UI 层

### 3. 主菜单新增“检查更新...”并复用同一动作

应用主菜单新增“检查更新...”项，并调用与偏好设置页按钮相同的更新动作。菜单项的可用状态也从 `UpdateManager` 获取，从而保持一致性。

备选方案：
- 只保留设置页入口
  - 放弃，因为桌面应用的更新入口应当在主菜单中可达，且用户已明确提出需求

### 4. 按架构拆分构建产物和 feed，而不是在单一 appcast 中混合多架构

发布链路改为：

- `arm64` 生成独立 DMG
- `x86_64` 生成独立 DMG
- 每个架构拥有自己的 feed
- 每个架构包写入自己的 `SUFeedURL`

这是最稳定的更新命中方式。客户端只消费一个架构专属 feed，避免 Sparkle 在同一 feed 中筛选多架构条目。

备选方案：
- 保持一个 `appcast.xml`，在同一 feed 中放多个架构产物
  - 放弃，因为筛选逻辑更复杂，排障更难

### 5. `appcast` 后处理必须保留 Sparkle 原始签名属性

`generate_appcast.sh` 仍可做 URL 改写和发布说明整理，但必须保留 Sparkle 原始生成的 `enclosure` 属性，尤其是签名相关字段。脚本输出后还要加入显式校验步骤，确认签名属性仍然存在。

备选方案：
- 重写整个 XML 节点，仅保留少量字段
  - 放弃，因为这正是当前更新失效的高风险来源

## Risks / Trade-offs

- [架构维度会让发布资产数量翻倍] → 通过固定命名约定和脚本参数把复杂度控制在发布脚本内部。
- [更新能力状态依赖 Sparkle 配置与 Bundle 元数据] → 把状态判断统一收敛到 `UpdateManager`，避免多处散落。
- [历史 `1.0` 安装包可能依然无法自动升级] → 接受一次性手动升级过渡，不伪装成可完全自动修复的问题。
- [发布脚本改动后可能引入新的人为操作错误] → 明确按架构的构建、发布、feed 生成顺序，并把验证作为发布步骤的一部分。

## Migration Plan

1. 先修改 OpenSpec 覆盖的更新能力与发布规范，明确应用内和发布链路的新要求。
2. 实现运行时更新入口统一、更新设置页联动和主菜单入口。
3. 实现按架构构建、按架构 feed 和 `appcast` 签名保留。
4. 用新链路发布下一个修复版，并分别验证 `arm64` 与 `x86_64` 包。

回滚策略：

- 如果运行时更新入口改造出现问题，可回滚应用内更新相关代码，不影响本地搜索功能。
- 如果分架构发布脚本出现问题，可回滚发布脚本与 feed 变更，临时恢复现有单包发布方式，同时停止自动更新发布。

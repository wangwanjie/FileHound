## Context

当前主搜索窗口的规则编辑器由 `SearchRuleCatalog`、`SearchRuleRowView`、`SearchRulesViewController` 和 `SearchFormViewController` 共同组成，但“字段定义”“操作符可用性”“值编辑器类型”“规则是否合法”分散在多个层次里：

- `SearchRuleCatalog` 只负责字段和操作符的粗粒度定义
- `SearchRuleRowView` 直接根据字段切换控件，但不理解支持状态
- `SearchFormViewController` 目前不会阻止无效规则触发搜索
- `SearchExecutor` 同时承担部分语义解释和一部分“其实不支持”的兜底行为

这导致 UI 容易暴露出“能选但不能用”的条件，而且日期条件与 `kind` 条件都无法对齐 FAF 风格交互。

## Goals / Non-Goals

**Goals:**

- 把字段、操作符、值编辑器、支持状态和阻塞原因收口成单一能力矩阵
- 让日期条件切换到 FAF 风格语义与编辑器
- 让 `kind` 支持 `is / is not + 类型下拉`，且底层比较稳定枚举而不是展示字符串
- 在搜索前统一校验规则，禁止不支持或无效组合进入执行层
- 保持当前规则保存、恢复和本地执行链路稳定

**Non-Goals:**

- 不在本轮实现 `comments` 和 `script` 的真实搜索能力
- 不为所有规则重写 Spotlight 查询生成逻辑
- 不重构 `QueryCompiler` 以覆盖全部 UI 可见规则
- 不追求 `kind` 对所有冷门文件格式做到完美识别

## Decisions

### 1. 用能力矩阵取代分散判断

核心决策是在 `SearchRuleCatalog` 侧建立统一能力矩阵，使其成为字段、操作符和值编辑器的单一真相来源。矩阵中每个字段都声明：

- 可见标题
- 值编辑器类型
- 可见操作符列表
- 每个操作符是否支持
- 不支持时的提示 key

这样 `SearchRuleRowView` 和 `SearchRulesViewController` 都不需要再自行推断“这个组合能不能用”。

备选方案：

- 继续在 UI 层按字段写 `switch`，执行层再兜底。
  - 放弃原因：现状已经证明这种方式会继续产生失配。

### 2. 将“暂不支持”和“用户输入无效”分开建模

规则状态拆成 `valid`、`unsupported`、`invalid` 三类：

- `unsupported` 表示产品尚未开放
- `invalid` 表示该字段已开放，但当前值不成立

这样可以清楚区分：

- `comments`、`script` 属于暂不支持
- `Kind is not any`、`is within the last` 未填值属于无效输入

备选方案：

- 全部视为统一错误文案。
  - 放弃原因：用户无法区分是产品没做，还是自己输入错了。

### 3. 日期条件按操作符切换编辑器，而不是沿用单一文本框

日期字段的交互改为操作符驱动：

- 绝对日期操作符使用日期输入
- 相对日期操作符使用“数值 + 单位”
- `today / yesterday` 不显示值输入

备选方案：

- 继续使用文本框，靠占位符提示格式。
  - 放弃原因：既不接近 FAF，也会继续放大无效输入问题。

### 4. `kind` 使用稳定枚举和归一化匹配

`kind` 不再复用字符串 contains/is 判断，而是新增内部稳定枚举 id：

- UI 下拉展示本地化文案
- 保存值使用稳定 id
- 执行层先把文件归一化到枚举，再做 `is / is not`

首版映射优先保证目录、包、链接、文本、图片、音视频、PDF、归档、应用等常见类型正确，其余类型回退到更宽泛大类。

备选方案：

- 直接对结果项里的 `kind` 文案做字符串比较。
  - 放弃原因：展示字符串会受语言切换和文案调整影响，不能作为稳定规则值。

### 5. 复用现有状态栏承载阻塞原因

主搜索窗口已有底部 `statusLabel`。本轮不新增弹窗或横幅，而是：

- 行内显示单条规则问题
- 底部状态栏显示首条阻塞原因
- `Find` 按钮禁用

备选方案：

- 点搜索时再弹 alert。
  - 放弃原因：打断式体验过重，而且不能在编辑阶段及时反馈。

## Risks / Trade-offs

- [`kind` 映射不够完整] → 首版先覆盖 FAF 列表里最常见类型，无法精确分类时回退到更宽泛大类，避免误判成错误细类。
- [日期相对时间测试脆弱] → 引入可注入时间源或集中封装当前时间读取，确保测试可冻结时间。
- [规则恢复后出现阻塞状态会暴露历史无效数据] → 明确保留原始选择并给出提示，不做静默修正。
- [能力矩阵扩展后代码路径变多] → 用视图测试和执行测试锁住字段切换、阻塞和执行语义。

## Migration Plan

1. 在 OpenSpec change 中落地 proposal/design/spec/tasks。
2. 在 `SearchRuleCatalog` 和规则 UI 层引入能力矩阵与校验结果。
3. 扩展日期与 `kind` 的执行语义，并补齐测试。
4. 接入主窗口阻塞反馈，验证无效规则不会发起搜索。

回滚策略：

- 回退本次 change 涉及的规则矩阵、UI 校验和执行语义改动，恢复到当前字段定义与执行行为。

## Open Questions

无。`kind` 的 UI 结构、`any` 的处理方式和暂不支持策略已在设计确认阶段定稿。

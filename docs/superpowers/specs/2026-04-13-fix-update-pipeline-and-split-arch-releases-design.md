# FileHound 更新链路修复与分架构发布设计

日期：2026-04-13  
状态：已确认，待写实现计划

## 1. 背景

FileHound 已经完成 `1.1.0` 发布，并具备 Sparkle 更新入口、DMG 打包脚本、GitHub Release 发布脚本和 `appcast.xml` 生成脚本。但这条发布与更新链路存在三个实质问题：

- 更新设置页中的“立即检查更新”按钮不可用
- 已安装 `1.0` 的应用无法稳定检查到 `1.1.0` 更新
- 当前发布仍按单一 Universal 包处理，不满足按 `arm64` / `x86_64` 分开打包和分开发布的要求

结合当前代码和已生成产物，已经确认以下风险点：

- 分发产物中的 `Info.plist` 未稳定包含 `SUFeedURL` 与 `SUPublicEDKey`
- `generate_appcast.sh` 的 XML 后处理阶段存在丢失 Sparkle `enclosure` 签名属性的风险
- 更新设置页与主菜单尚未形成统一的更新触发入口
- 现有发布脚本默认构建 Universal 包，没有架构维度的产物、feed 和发布策略

## 2. 目标

- 修复 Sparkle 更新链路，使正式发布包能够稳定执行手动检查更新与启动时检查更新
- 在主菜单中补充“检查更新...”入口，并与设置页共用同一套更新逻辑
- 将正式分发流程改为 `arm64` 与 `x86_64` 分架构打包、分架构发布、分架构 feed
- 让后续版本在 Sparkle 层面只命中当前架构对应的更新源，避免跨架构更新
- 与 OpenSpec 现有 [release-distribution-pipeline 规范](/Users/VanJay/Documents/Work/Private/FileHound/openspec/specs/release-distribution-pipeline/spec.md) 保持一致，并为后续变更扩展留下明确边界

## 3. 非目标

- 不尝试修复已经发布且缺失必要 Sparkle 元数据的历史安装包内部逻辑
- 不在本设计内引入 GitHub Actions 自动发布
- 不恢复“机台氛围”相关功能需求，该项已由用户明确排除
- 不在本设计内处理与更新链路无关的偏好设置 UI 微调

## 4. 方案对比

### 4.1 方案 A：单一 appcast 承载双架构产物

做法：

- 保持一个 `appcast.xml`
- 在同一 feed 中维护 `arm64` 与 `x86_64` 两类产物
- App 继续只持有一个 `SUFeedURL`

优点：

- 发布入口最少
- 表面上对现有脚本改动较少

缺点：

- Sparkle 对多架构候选项的筛选更依赖 feed 元数据细节
- 出现误命中或未命中时排查成本高
- 不利于后续按架构独立回滚或重新发布

### 4.2 方案 B：按架构拆分 feed

做法：

- `arm64` 与 `x86_64` 各自产出独立 DMG
- 各自维护独立 appcast
- 构建时分别写入对应架构的 `SUFeedURL`

优点：

- 更新命中逻辑最直接、最稳定
- 发布、回滚、排障都能按架构隔离
- 与“分架构打包”的需求完全一致

缺点：

- 发布产物和脚本参数会增加一层架构维度

### 4.3 方案 C：先只修更新链路，继续发布 Universal 包

做法：

- 仅修复 `Info.plist`、`appcast` 签名、更新按钮和菜单入口
- 暂不改双架构打包

优点：

- 上线速度最快

缺点：

- 不满足当前明确需求
- 后续还需要二次改造发布链

### 4.4 推荐方案

采用方案 B。

原因：

- 它同时满足“修更新”和“分架构发布”两个目标
- 客户端只消费本架构 feed，行为最可预测
- 脚本复杂度虽然略有增加，但远低于后续排障成本

## 5. 设计

### 5.1 更新链路修复

更新链路修复分为四层。

第一层是分发包元数据：

- 保证最终 Release App 包内稳定存在 `SUFeedURL` 与 `SUPublicEDKey`
- 不能只依赖工程配置中声明了这些值，还必须验证归档产物中实际存在

第二层是 appcast 可信性：

- `generate_appcast.sh` 必须保留 Sparkle 原始生成的 `enclosure` 签名属性
- 后处理只能改写 URL、发布说明或展示性字段，不能破坏 Sparkle 用于校验的签名信息

第三层是运行时更新入口：

- `UpdateManager` 作为唯一更新协调者
- 设置页“立即检查更新”和主菜单“检查更新...”都直接走同一入口
- `canCheckForUpdates` 需要基于实际可用条件计算，而不是停留在静态 UI 层面

第四层是启动时自动检查：

- 应用设置中的更新策略必须同步到 Sparkle updater 配置
- “启动时检查更新”不能只是应用启动后调用一次方法，还要保证 Sparkle 本身处于有效配置状态

### 5.2 更新 UI 与主菜单

设置页与主菜单要共享一个事实来源：

- 设置页按钮的启用状态由 `UpdateManager` 统一提供
- 工具提示要能解释为什么当前不可检查更新，例如缺少 feed 元数据
- 主菜单新增“检查更新...”项，行为与设置页一致
- 菜单项和按钮都不自行复制检查逻辑，避免两处行为漂移

这样可以解决“设置页按钮灰掉，但用户不知道原因”和“菜单没有同等入口”的问题。

### 5.3 分架构打包

打包脚本改为显式按架构构建：

- `build_dmg.sh --arch arm64`
- `build_dmg.sh --arch x86_64`

每次只归档一个架构，不再默认产出 Universal 包。每个架构产物都带上架构后缀，例如：

- `FileHound_v1.1.1_3_arm64.dmg`
- `FileHound_v1.1.1_3_x86_64.dmg`

这样可以让产物命名、归档目录、发布资产名和 appcast 条目天然一一对应。

### 5.4 分架构 feed

每个架构拥有独立 feed：

- `appcast-arm64.xml`
- `appcast-x86_64.xml`

构建 `arm64` 包时写入 `arm64` feed URL，构建 `x86_64` 包时写入 `x86_64` feed URL。  
这样客户端永远只看自己架构对应的更新流，不需要在同一 feed 中做复杂筛选。

### 5.5 发布脚本与资产组织

发布脚本应一次性支持双架构发布，但逻辑上仍保持分步：

1. 分别构建两个架构的 DMG
2. 分别归档并生成两个架构的 appcast
3. 将两个 DMG 与两个 appcast 发布到同一版本 Release

这样保留原有“分步、可人工插入检查”的发布风格，同时满足新需求。

### 5.6 历史版本兼容策略

对已经安装 `1.0` 但未收到 `1.1.0` 更新的用户，要明确区分两类情况：

- 如果旧包本身缺少有效 `SUFeedURL` / `SUPublicEDKey`，则它无法完全依赖服务器侧修复恢复更新能力
- 从修复后的新版本开始，后续所有版本升级链路将恢复正常

因此兼容策略是：

- 从下一个修复版起，所有新安装和后续升级都走分架构 Sparkle 更新
- 对历史 `1.0` 用户准备一次手动升级说明，作为过渡

## 6. OpenSpec 对齐

本设计直接对应现有 [release-distribution-pipeline 规范](/Users/VanJay/Documents/Work/Private/FileHound/openspec/specs/release-distribution-pipeline/spec.md) 的三个方向：

- 正式 DMG 发布步骤
- GitHub Release 与 appcast 分发
- 分发包中的 Sparkle feed metadata

但当前基础规范还不足以覆盖以下新增要求：

- 更新设置页与主菜单的统一更新入口
- 启动时检查更新的运行时联动
- `arm64` / `x86_64` 分架构 feed 与分架构产物发布

因此后续实现前，应通过新的 OpenSpec change 扩展现有发布规范，而不是直接越过变更流程修改实现。

## 7. 验证策略

### 7.1 构建与产物验证

- `arm64` 与 `x86_64` 各自构建一份 Release 包
- 检查最终 `Info.plist` 是否包含正确的 `SUFeedURL` 和 `SUPublicEDKey`
- 检查两个 appcast 的 `enclosure` 是否保留 Sparkle 签名字段
- 检查资产 URL、版本号、构建号、文件大小与架构后缀是否一致

### 7.2 应用内行为验证

- 设置页“立即检查更新”可点击
- 主菜单“检查更新...”可点击
- 两者均通过同一 `UpdateManager` 触发更新检查
- “启动时检查更新”开启后，应用启动会触发更新检查
- “仅手动检查”模式下，启动不自动触发

### 7.3 版本兼容验证

- 用修复前但具备有效 Sparkle 配置的旧版本验证升级链路
- 对 `1.0` 历史包确认其无法自动升级的真实原因，并在发布说明中明确说明

## 8. 风险与缓解

- 风险：双架构 feed 引入更多发布文件  
  缓解：通过统一命名规则和脚本参数控制复杂度

- 风险：脚本后处理再次破坏 Sparkle 必需字段  
  缓解：把签名字段保留作为 appcast 生成后的显式验证步骤

- 风险：设置页、菜单、启动时检查三处状态不一致  
  缓解：所有入口统一由 `UpdateManager` 暴露状态与动作

- 风险：历史 `1.0` 用户期望无缝自动更新  
  缓解：明确标注一次性手动升级策略，不把不可恢复的问题伪装成可修复问题

## 9. 完成标准

- 最终分发包稳定包含 Sparkle 所需元数据
- `appcast` 保留 Sparkle 签名并可被正常消费
- 设置页和主菜单都可执行“检查更新”
- 启动时检查更新策略生效
- 正式发布链路支持 `arm64` 与 `x86_64` 分架构构建、分架构 feed 与分架构资产
- 后续实现通过 OpenSpec change 承接，而不是绕过规范直接改实现
